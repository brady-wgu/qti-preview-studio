>   and formatting the output.
> - QTIv12_test_bank_items.zip: This file provides an exemplar of an input
>   file conforming to the QTI v1.2 specification. It contains several item
>   types, including some that utilize image assets. Assets are contained in
>   the Resources folder, while the imsmanifest.xml and the various item xml
>   files are on the root of the QTI archive file.
> - QTIv21_test_bank_items.zip: This file provides an exemplar of an input
>   file conforming to the QTI v2.1 specification. For reasons I am hoping we
>   don't need to explore at present, this QTI file contains only multiple
>   choice item types, including some that utilize image assets. Assets are
>   contained in the Resources folder, while the imsmanifest.xml and the
>   various item xml files are on the root of the QTI archive file. While the
>   exemplar contains only multiple choice items, the hope is that the
>   application we are creating will accommodate the full range of item types
>   represented by the v1.2 QTI file provided.
>
> Our goal is to create a lightweight application that can process QTI input
> files and generate HTML representations of the assessment items contained
> in those files in two formats, one that approximates the online testing
> experience where the items are presented sequentially, one at a time as a
> kind of slideshow that the user can advance forward and backwards through
> the items list, and another format where the items are rendered in a
> vertical page format, one item after another with a lightweight visual
> indicator, like a horizontal rule separating them that is more suitable for
> a quick scan or for printing an answer key for the items.
>
> Processing QTI files into HTML will help my team work around the
> shortcomings of existing assessment item authoring and delivery platforms
> and serve two key functions:
> 1. It will allow us to preview the contents of a QTI file in a format that
>    approximates the ultimate delivery formatting before we ingest them into
>    the platform. This should help us identify and address gross errors
>    before importing items and reduce subsequent QA and clean up time.
> 2. It will provide a static visual representation of the items in a QTI
>    file, in a format that approximates the ultimate delivery formatting,
>    that will serve as a visual reference useful for validating that the
>    items were successfully imported into the item bank without losing
>    formatting or fidelity in the QTI transport process.
>
> To go a little deeper, the user of the app needs to be able to do the
> following:
> 1. Launch the app.
> 2. Select one or more QTI files to process (compressed zip file) that
>    conform to the v1.2 or v2.1 versions of that specification.
> 3. Process the selected QTI file(s) to generate the required output. For
>    each processed QTI file the required outputs are:
>    1. An output folder should be created for each input file, named after
>       that input file without the filetype extension.
>    2. Within each output folder should be placed the following files
>       generated from the input QTI:
>       1. ITEM_PREVIEW.HTML -- [full "online test" experience requirements:
>          item ID/label/code/shuffle indicator displayed above each item;
>          all item components present and correctly formatted; correct
>          response indicated via bold and underline; images correctly
>          presented; advance/back UI derived from the formatting files;
>          styling derived from the formatting files; items ordered
>          ascending alphanumeric by identifier]
>       2. PRINT_PREVIEW.HTML -- [same content/formatting requirements as
>          ITEM_PREVIEW.HTML, but as one scrolling document with a light
>          horizontal rule between items instead of slideshow navigation]
>       3. RESOURCES folder -- item assets plus any other supporting files.
>       4. Log.txt -- summary of items converted, issues encountered, and
>          enough detail to troubleshoot problems.
> 4. View the applied stylesheet data for the items (derived from the
>    formatting files).
> 5. Manually adjust the stylesheet data to dial in formatting of the items
>    in the HTML output. User adjustments should persist until reset.
> 6. Reset the applied stylesheet data to the baseline stylesheet data
>    derived from the formatting files.
> 7. The application should work offline, and it should not send data to
>    another party or otherwise produce output other than what is specified
>    here.
> 8. To the extent possible, the user experience and output files should not
>    vary based on the QTI specification of the input files, i.e. we want the
>    output to be the same regardless of the input in the encoding.
>
> Ultimately, I'd like to produce the following deliverables for this tool:
> - A MacOSX standalone binary that can be run as an application.
> - A Windows standalone binary that can be run as an application.
> - The source files for each (if different), as well as any necessary
>   supporting files to build or compile the binaries. [...] Where design or
>   coding decisions were made based on the inputs of this prompt, please
>   annotate in the source files via inline comments [...] Developer
>   documentation of the source files [...]
> - A verbatim transcript of the prompt session [...]
> - A version of the prompt session that created the app, optimized for a
>   future Claude (or other LLM [...]) that a user could paste in to a prompt
>   to accurately reproduce these outputs.
> - A readme.txt that summarizes the uses of the app, and describes all of
>   the output files.
> - A brief instructional document [...] Quick start guide for users [...]
>
> As far as order of operations, I was thinking the following would work,
> but feel free to suggest other, more efficient approaches where
> applicable:
> 1. You parse this prompt and the provided input files.
> 2. We'll work through any clarifications you need [...]
> 3. Next we'll iterate together to dial in the two HTML experiences [...]
> 4. Once we have the UI/UX of the output files dialed in, I believe it is

>
> So let's start there - here are the files. Let me know when you are ready

[Claude discovered the "no correct answer visible" bug in item #1 was
actually the SAME root cause as item #4's image-marking gap: the underlying
data was correct, but bold/underline CSS has no visible effect on an `<img>`
element. Claude fixed this with a treatment that works for any content type
(green row highlight + checkmark badge, in addition to bold/underline for
text), moved FIB/dropdown answers to a below-item "Answer Key" box, redrew
the ITEM_PREVIEW chrome closer to the source platform (navy header, fixed
bottom nav strip), and bundled the user's own exact KaTeX/auto-render/CSS
files (pulled from their formatting zip) supplemented with standard KaTeX
font files, with a live toggle button to compare rendered vs. raw LaTeX.]
